## Analysis of Time Complexity and Big-O Notation

### 1. Introduction to Time Complexity

Time complexity is a measure of the amount of time taken by an algorithm to run as a function of the length of its input. It quantifies how the runtime of an algorithm grows as the input size increases, allowing for the comparison of algorithm efficiency independently of specific hardware or implementation details. Instead of measuring actual execution time in seconds, which can vary widely, we count the number of basic operations an algorithm performs.

### 2. Big-O Notation

Big-O notation is a mathematical notation that describes the limiting behavior of a function when the argument tends towards a particular value or infinity. In computer science, it is used to classify algorithms according to how their running time or space requirements grow as the input size grows. It provides an upper bound on the growth rate of the function, effectively representing the worst-case scenario.

Common Big-O complexities include:
*   **O(1)**: Constant time (e.g., accessing an array element by index).
*   **O(log n)**: Logarithmic time (e.g., binary search).
*   **O(n)**: Linear time (e.g., traversing a list).
*   **O(n log n)**: Linearithmic time (e.g., merge sort, heap sort).
*   **O(n^2)**: Quadratic time (e.g., bubble sort, nested loops iterating n times).
*   **O(n^k)**: Polynomial time (where k is a constant).
*   **O(2^n)**: Exponential time (e.g., some recursive algorithms without memoization).
*   **O(n!)**: Factorial time (e.g., brute-force solution to Traveling Salesperson Problem).

When determining Big-O, we typically:
1.  **Count fundamental operations**: Assignments, arithmetic operations, comparisons, array accesses, function calls.
2.  **Identify the dominant term**: In a polynomial expression of operations, the term with the highest power of `n` (the input size) dictates the growth rate for large `n`.
3.  **Discard constant factors**: Constants do not significantly affect the growth rate for large `n`. For example, `2n` and `5n` both simplify to `O(n)`.
4.  **Discard lower-order terms**: As `n` grows large, lower-order terms become insignificant compared to the dominant term. For instance, `n^2 + n + 1` simplifies to `O(n^2)`.

### 3. Analysis Methodology

For each pseudo-code snippet, the analysis will proceed as follows:
1.  Present the pseudo-code.
2.  Break down the code into its constituent operations or blocks.
3.  Determine the number of times each operation or block executes in relation to the input size `n`.
4.  Sum the operation counts to obtain an overall complexity function.
5.  Apply Big-O rules (dominant term, discarding constants) to derive the final Big-O notation.

#### 4.1. Snippet 1: Constant Time (O(1))

**Pseudo-code:**
```pseudo
Function calculate_fixed_sum(a, b):
  1. result = a + b
  2. print(result)
End Function
```

**Analysis:**
1.  **Line 1 (`result = a + b`):** This line involves one arithmetic operation (addition) and one assignment operation. These are constant-time operations, regardless of the values of `a` and `b`. Let's denote this as $C_1$ operations.
2.  **Line 2 (`print(result)`):** This line involves a single output operation. This is also a constant-time operation. Let's denote this as $C_2$ operations.

**Total Operations:** The total number of operations is $C_1 + C_2$. Since $C_1$ and $C_2$ are fixed constants, their sum is also a constant, say $C$.

**Big-O Notation:** $O(1)$
This algorithm's execution time does not depend on the magnitude of `a` or `b` or any input size `n` (as there is no `n` here). It performs a fixed number of operations.

#### 4.2. Snippet 2: Linear Time (O(n))

**Pseudo-code:**
```pseudo
Function find_max(array, n):
  1. max_val = array[0]
  2. For i from 1 to n-1:
  3.   If array[i] > max_val:
  4.     max_val = array[i]
  5.   End If
  6. End For
  7. Return max_val
End Function
```
*Assumption: `n` is the number of elements in `array`.*

**Analysis:**
1.  **Line 1 (`max_val = array[0]`):** This involves one array access and one assignment. These are constant-time operations. Let's say $C_1$ operations.
2.  **Line 2-6 (`For i from 1 to n-1` loop):** This `for` loop iterates `n-1` times.
    *   **Loop Initialization:** The loop variable `i` is initialized (e.g., `i = 1`), which is a constant operation, say $C_2$.
    *   **Loop Condition Check:** The condition `i <= n-1` is checked `n` times (once before each iteration and once after the last iteration fails). Each check is a constant operation, say $C_3$.
    *   **Loop Variable Increment:** The variable `i` is incremented `n-1` times. Each increment is a constant operation, say $C_4$.
    *   **Lines 3-5 (Inner operations):** Inside the loop, for each iteration:
        *   `array[i] > max_val`: One array access and one comparison. Constant operations, say $C_5$.
        *   `max_val = array[i]`: One array access and one assignment (only if condition is true, but we consider the worst-case, so it could happen every time). Constant operations, say $C_6$.
        *   Thus, inside the loop, we have at most $C_5 + C_6$ constant operations per iteration.
    *   Total operations for the loop: $C_2 + n \cdot C_3 + (n-1) \cdot C_4 + (n-1) \cdot (C_5 + C_6)$.
3.  **Line 7 (`Return max_val`):** A single return operation. Constant time, $C_7$.

**Total Operations:**
Summing these up, the total number of operations $T(n)$ is approximately:
$T(n) = C_1 + (C_2 + n \cdot C_3 + (n-1) \cdot C_4 + (n-1) \cdot (C_5 + C_6)) + C_7$
$T(n) = C_1 + C_2 + C_7 + C_3 n + C_4 (n-1) + (C_5+C_6)(n-1)$
$T(n) = C_{total\_constants} + (C_3 + C_4 + C_5 + C_6)n - (C_4 + C_5 + C_6)$
$T(n) = A \cdot n + B$, where $A$ and $B$ are constants.

**Big-O Notation:** $O(n)$
The dominant term is $A \cdot n$. As `n` grows, the linear term dictates the growth rate.

#### 4.3. Snippet 3: Quadratic Time (O(n^2))

**Pseudo-code:**
```pseudo
Function print_all_pairs(array, n):
  1. For i from 0 to n-1:
  2.   For j from 0 to n-1:
  3.     print(array[i], array[j])
  4.   End For
  5. End For
End Function
```
*Assumption: `n` is the number of elements in `array`.*

**Analysis:**
1.  **Line 1 (`For i from 0 to n-1`):** This is the outer loop, which iterates `n` times.
    *   Initialization, condition checks, and increments for `i` take approximately constant operations per iteration, summing to $C_1 \cdot n + C_2$ overall for the outer loop's overhead.
2.  **Line 2 (`For j from 0 to n-1`):** This is the inner loop. For *each* iteration of the outer loop, this inner loop also iterates `n` times.
    *   Initialization, condition checks, and increments for `j` take approximately constant operations per iteration, summing to $C_3 \cdot n + C_4$ for each *outer loop* iteration.
3.  **Line 3 (`print(array[i], array[j])`):** This line, involving two array accesses and a print operation, is a constant-time operation, say $C_5$. This operation is executed inside the inner loop.

**Total Operations:**
*   The outer loop runs `n` times.
*   For each of these `n` times, the inner loop runs `n` times.
*   Inside the inner loop, $C_5$ operations are performed.
*   So, the total number of `print` operations (and associated array accesses) is $n \cdot n \cdot C_5 = C_5 n^2$.
*   Adding the loop overheads, $T(n) = (C_1 n + C_2) + n \cdot (C_3 n + C_4 + C_5)$.
*   $T(n) = C_1 n + C_2 + C_3 n^2 + C_4 n + C_5 n^2$
*   $T(n) = (C_3 + C_5)n^2 + (C_1 + C_4)n + C_2$
*   $T(n) = A \cdot n^2 + B \cdot n + D$, where $A, B, D$ are constants.

**Big-O Notation:** $O(n^2)$
The dominant term is $A \cdot n^2$. The execution time grows quadratically with the input size.

#### 4.4. Snippet 4: Logarithmic Time (O(log n))

**Pseudo-code:**
```pseudo
Function binary_search(sorted_array, target, n):
  1. low = 0
  2. high = n - 1
  3. While low <= high:
  4.   mid = floor((low + high) / 2)
  5.   If sorted_array[mid] == target:
  6.     Return mid
  7.   Else If sorted_array[mid] < target:
  8.     low = mid + 1
  9.   Else:
  10.    high = mid - 1
  11.  End If
  12. End While
  13. Return -1
End Function
```
*Assumption: `n` is the number of elements in `sorted_array`.*

**Analysis:**
1.  **Lines 1-2 (`low = 0`, `high = n - 1`):** These are initial assignments, taking constant time. Let this be $C_1$.
2.  **Lines 3-12 (`While low <= high` loop):** This loop continues as long as the search space is valid.
    *   Inside the loop, all operations (midpoint calculation, array access, comparisons, assignments to `low` or `high`) are constant-time operations. Let these be $C_2$ operations per iteration.
    *   The crucial aspect is how many times this loop executes. In each iteration, the search space (defined by `high - low + 1`) is approximately halved.
        *   Initial range size: `n`.
        *   After 1 iteration: `n/2`.
        *   After 2 iterations: `n/4`.
        *   After `k` iterations: `n / 2^k`.
    *   The loop terminates when the search space becomes 1 or 0 (i.e., `low > high`). This happens when `n / 2^k` is approximately 1.
    *   $n / 2^k \approx 1 \implies n \approx 2^k \implies k \approx \log_2 n$.

**Total Operations:**
The number of iterations is proportional to $\log_2 n$. Since each iteration performs a constant number of operations ($C_2$), the total operations are approximately $C_1 + C_2 \cdot \log_2 n$.

**Big-O Notation:** $O(\log n)$
The dominant term is $C_2 \cdot \log_2 n$. The base of the logarithm is usually ignored in Big-O notation as it only changes by a constant factor ($\log_a n = \log_b n / \log_b a$).

#### 4.5. Snippet 5: Linearithmic Time (O(n log n))

**Pseudo-code:**
```pseudo
Function analyze_elements_with_log_search(array, n):
  1. For i from 0 to n-1: // Outer loop iterates n times
  2.   // Inner "search" loop that halves its search space
  3.   low = 0
  4.   high = n - 1
  5.   While low <= high:
  6.     mid = floor((low + high) / 2)
  7.     // Some constant operations, e.g., comparing array[mid] with array[i]
  8.     If array[mid] == array[i]:
  9.       Break
  10.    Else If array[mid] < array[i]:
  11.      low = mid + 1
  12.    Else:
  13.      high = mid - 1
  14.    End If
  15.  End While
  16. End For
End Function
```
*Assumption: `n` is the number of elements in `array`.*

**Analysis:**
1.  **Line 1 (`For i from 0 to n-1`):** This is the outer loop, which iterates `n` times.
    *   The overhead for this loop (initialization, condition checks, increments for `i`) contributes $C_1 \cdot n + C_2$ to the total time.
2.  **Lines 3-15 (Inner `While` loop block):** This entire block of code is executed for each of the `n` iterations of the outer loop.
    *   Within this block (Lines 3-15), we observe a pattern identical to binary search (Snippet 4).
    *   **Lines 3-4 (`low = 0`, `high = n - 1`):** Constant-time initializations ($C_3$).
    *   **Lines 5-14 (`While low <= high`):** This loop performs a constant number of operations ($C_4$) in each iteration, and its number of iterations is proportional to $\log_2 n$. So, this inner block (Lines 3-15) takes approximately $C_3 + C_4 \cdot \log_2 n$ time.

**Total Operations:**
*   The outer loop runs `n` times.
*   For each iteration of the outer loop, the inner block (which is effectively a binary search on `n` elements) takes $O(\log n)$ time.
*   Therefore, the total number of operations $T(n)$ is approximately:
    $T(n) = (C_1 n + C_2) + n \cdot (C_3 + C_4 \log_2 n)$
    $T(n) = C_1 n + C_2 + C_3 n + C_4 n \log_2 n$
    $T(n) = C_4 n \log_2 n + (C_1 + C_3)n + C_2$
    $T(n) = A \cdot n \log n + B \cdot n + D$, where $A, B, D$ are constants.

**Big-O Notation:** $O(n \log n)$
The dominant term is $A \cdot n \log n$.

#### 4.6. Snippet 6: Conditional Logic and Function Calls (O(n^2))

**Pseudo-code:**
```pseudo
Function process_data_complex(data_array, n):
  1. total_sum = 0
  2. For i from 0 to n-1:
  3.   total_sum = total_sum + data_array[i] // O(1) per iteration
  4. End For
  5. If total_sum > 1000:
  6.   result = calculate_expensive_value(n) // This function is O(n^2)
  7. Else:
  8.   result = calculate_cheap_value(n) // This function is O(n)
  9. End If
  10. Return result
End Function

Function calculate_expensive_value(size):
  1. complex_result = 0
  2. For x from 0 to size-1:
  3.   For y from 0 to size-1:
  4.     complex_result = complex_result + x * y // O(1)
  5.   End For
  6. End For
  7. Return complex_result
End Function

Function calculate_cheap_value(size):
  1. simple_result = 0
  2. For z from 0 to size-1:
  3.   simple_result = simple_result + z // O(1)
  4. End For
  5. Return simple_result
End Function
```
*Assumption: `n` is the number of elements in `data_array`.*

**Analysis:**

First, analyze the helper functions:
**`calculate_expensive_value(size)`:**
*   Lines 2-6: Nested loops. The outer loop runs `size` times. The inner loop runs `size` times for each outer loop iteration.
*   Line 4 (`complex_result = complex_result + x * y`): Constant operations.
*   Total for `calculate_expensive_value(size)`: $C_A \cdot \text{size}^2$.
*   **Big-O:** $O(\text{size}^2)$, which is $O(n^2)$ when `size` is `n`.

**`calculate_cheap_value(size)`:**
*   Lines 2-4: Single loop. The loop runs `size` times.
*   Line 3 (`simple_result = simple_result + z`): Constant operations.
*   Total for `calculate_cheap_value(size)`: $C_B \cdot \text{size}$.
*   **Big-O:** $O(\text{size})$, which is $O(n)$ when `size` is `n`.

Now, analyze `process_data_complex(data_array, n)`:
1.  **Line 1 (`total_sum = 0`):** Constant time ($C_1$).
2.  **Lines 2-4 (`For i from 0 to n-1`):** A single loop iterating `n` times.
    *   Line 3 (`total_sum = total_sum + data_array[i]`): Constant operations.
    *   Total for this loop: $C_2 \cdot n$.
    *   **Big-O:** $O(n)$.
3.  **Lines 5-9 (`If total_sum > 1000 ... End If`):** This is a conditional block.
    *   **Condition Check:** `total_sum > 1000` is a constant-time comparison ($C_3$).
    *   **`If` block (Line 6):** Calls `calculate_expensive_value(n)`. We determined this takes $O(n^2)$ time.
    *   **`Else` block (Line 8):** Calls `calculate_cheap_value(n)`. We determined this takes $O(n)$ time.
    *   When analyzing conditional statements, we consider the worst-case scenario. The worst case is the branch that executes the most operations. Here, `O(n^2)` is asymptotically worse than `O(n)`.
    *   Therefore, the complexity of the entire `If/Else` block is $\max(O(n^2), O(n)) = O(n^2)$.
4.  **Line 10 (`Return result`):** Constant time ($C_4$).

**Total Operations:**
Summing the complexities:
$T(n) = C_1 + O(n) + O(n^2) + C_4$
$T(n) = \text{constant} + \text{linear term} + \text{quadratic term}$
$T(n) = O(n^2) + O(n) + O(1)$

**Big-O Notation:** $O(n^2)$
The dominant term is $O(n^2)$. Even though there's an $O(n)$ part at the beginning, for large `n`, the quadratic complexity of the `if` branch (worst case) will dominate the overall runtime.
